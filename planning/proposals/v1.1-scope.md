### CO-1 — `session_start_price` missing from SSE payload

| Field | Value |
|---|---|
| Source | PLAN.md §6 + `backend/app/market/models.py` |
| Priority | **P0** |
| Effort | **S** |

**Evidence of gap:**  
PLAN.md §6 specifies: *"Each SSE event contains ticker, price, previous price, **session-start price**, timestamp, and change direction."* and *"'Daily change %' displayed in the UI is computed from `(price - session_start_price) / session_start_price * 100`."*

The committed `PriceUpdate` dataclass (`backend/app/market/models.py`, lines 10–49) has no `session_start_price` field. The `to_dict()` output (lines 39–49) does not include it. `PriceCache.update()` (`cache.py`, lines 23–42) sets `previous_price` from the last update only — it does not persist the first-seen price per ticker.

**Impact:** The frontend cannot compute "daily change %" as specified. The watchlist panel (PLAN.md §10) is blocked on this field.

**Evidence of done:**  
- `PriceUpdate` has a `session_start_price: float` field populated on first cache entry for a ticker
- `to_dict()` includes `session_start_price` in its output
- `PriceCache.update()` stores and preserves the session-start price (only set once, never overwritten)
- All existing `test_cache.py` and `test_models.py` tests still pass; new tests cover session-start behavior

## §6 — Recommended v1.1 Milestone Shape

### Proposed phase breakdown (7 phases)

| Phase | Name | Items | Effort | Priority |
|---|---|---|---|---|
| 0 | Market data carry-overs | CO-1 (session_start_price), CO-2, CO-3, CO-4, CO-5 | S | P0/P2 |
| 1 | Database + REST API | N-1, N-2 | M+M | P0 |
| 2 | LLM integration | N-3 | M | P1 |
| 3 | Frontend core | N-4 | L | P0 |
| 4 | Frontend portfolio views | N-5 | M | P1 |
| 5 | AI chat UI | N-6 | M | P1 |
| 6 | Docker + scripts | N-7 | S+M | P1 |
| 7 | E2E tests | N-8 | M | P2 |

**Total phases:** 8 (Phase 0 through Phase 7)  
**Sequencing rationale:**

- Phase 0 must precede Phase 3: the frontend SSE consumer needs `session_start_price` in the payload (CO-1). The other CO items are independent quality work.
- Phase 1 must precede Phase 2 and Phase 3: API routes require the database; the frontend needs the REST API.
- Phase 2 can parallel Phase 3 once Phase 1 is done (backend chat endpoint and frontend core are independent).
- Phase 4 and Phase 5 follow Phase 3 (portfolio views and chat UI require the frontend shell).
- Phase 6 (Docker) can begin structurally once Phase 1 is done but only fully completes once Phase 3 (frontend build) exists.
- Phase 7 (E2E) is the last gate — requires the full stack running in Docker.

### Suggested sequencing

```
Phase 0  (S)  ──► Phase 1  (M+M)  ──► Phase 2  (M)  ──► Phase 5  (M)
                        │                                     │
                        └──────────► Phase 3  (L)  ──► Phase 4  (M)
                                          │                   │
                        Phase 6  (S+M) ◄──┘                  │
                              │                               │
                              └───────────► Phase 7  (M) ◄───┘
```

## §7 — Open Questions

1. **Phase numbering convention:** This proposal introduces a new Phase 0–7 numbering scheme for v1.1. The triage prompt referenced "Phase 7" and "Phase 9" with no prior definition in any repo document. Should the team establish a canonical phase-numbering convention in a PROJECT.md or roadmap file before v1.1 build begins?

2. **v1.0 milestone-close artifacts:** No audit, retrospective, or milestone-close document was found for v1.0. Is this a process gap to close (create a lightweight template) or a deliberate choice to keep overhead low?

3. **CO-1 priority trade-off:** `session_start_price` (CO-1) could be implemented as a frontend approximation instead — the client tracks the first price it receives over SSE since page load. This avoids modifying the backend data model but means "daily change %" resets on page reload. Is the backend-authoritative approach (per PLAN.md §6) required, or is the client-side approximation acceptable?

4. **Windows script testing:** The task mentioned `start_windows.ps1` / `stop_windows.ps1` as items with "gaps." Since these scripts don't yet exist, will Windows testing be manual (human runs scripts on Windows Docker Desktop) or automated (CI matrix)? This affects the evidence-of-done for N-7/Phase 6.

5. **Cross-arch Docker buildx:** PLAN.md §11 does not mention multi-arch builds. The triage prompt referenced `docker buildx` as a Phase 9 item, but no source document describes the requirement. Should `linux/amd64` + `linux/arm64` be an explicit requirement for v1.1, and if so, against what CI runner?

6. **Agent parallelism:** Phases 2 (LLM integration) and 3 (frontend core) are independent once Phase 1 completes. Should these be handed to separate agents running in parallel to reduce wall-clock time?
