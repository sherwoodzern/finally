### Reference test body (verbatim from RESEARCH.md §"SSE Reconnect Test Pattern" lines 737-768)

```typescript
import { test, expect } from '@playwright/test';

test('SSE reconnect: dot flips to reconnecting/disconnected on abort, returns to connected on unroute', async ({ page, context }) => {
  // 1. Confirm initial connected state.
  await page.goto('/');
  await expect(page.getByLabel('SSE connected')).toBeVisible({ timeout: 10_000 });

  // 2. Arm the abort. context.route covers ALL future requests including
  //    the EventSource's auto-retries — survives page navigation.
  //    route.abort('connectionreset') simulates a TCP RST. We are NOT
  //    fulfilling SSE — Playwright issue #15353 does not apply.
  await context.route('**/api/stream/prices', (route) =>
    route.abort('connectionreset'),
  );

  // 3. Trigger a reconnect cycle. page.reload() forces the EventSource to
  //    re-create; the new connection hits the abort and the connection-status
  //    state machine walks to 'reconnecting' / 'disconnected'.
  await page.reload();

  await expect(
    page.getByLabel(/^SSE (reconnecting|disconnected)$/),
  ).toBeVisible({ timeout: 15_000 });

  // 4. Lift the abort. Subsequent EventSource retries succeed (server
  //    retry: 1000 gives a known 1s reconnect floor). EventSource backs off
  //    exponentially after multiple failures, so the timeout is generous —
  //    especially for WebKit (RESEARCH.md WebKit caveat lines 781-783).
  await context.unroute('**/api/stream/prices');

  await expect(page.getByLabel('SSE connected')).toBeVisible({ timeout: 20_000 });
});
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test browser ↔ EventSource ↔ network ↔ FastAPI SSE | The four-hop chain that 07-sse-reconnect exercises. The route handler intercepts at the first hop (browser-to-network); the EventSource is the consumer; FastAPI's SSE router is the producer. The test simulates a network failure between hops 2 and 3. |
| Playwright route handler ↔ EventSource auto-retry | EventSource follows the WHATWG spec — on error, fires `error` event, waits the `retry:` interval (1s from server), reconnects. Three browsers implement this slightly differently (Chromium fires immediately; Firefox sometimes batches; WebKit can take 1-2s longer). The test's regex selectors and 15-20s timeouts absorb the variability. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-10-05-02 | I (Information disclosure) | Aborted SSE traces in test reports | accept | A failed reconnect produces a Playwright trace that may include the URL `/api/stream/prices` and `connectionreset` error codes. No PII; no secrets. Reports gitignored. |
